大家好,今天我们来简单学习一下PubSub模式,了解一下Nodejs下广泛使用的EventEmitter对象,以及简单实现一个PubSub类.

​ 使用PubSub模式在某种程度上可以让我们的应用易于扩展,低耦合且维护性高的代码.我们可以很方便的添加额外的功能.💪

PubSub 模式

在早期的JavaScript在浏览器端对于DOM元素附加事件处理器的业务中,也许我们能看到如下代码片段:

// 添加一个事件监听
2div.onclick = clickHandler;
3// 添加多个
4div.onclick = function() {
5  clickHandler1.apply(this, arguments)
6  clickHandler2.apply(this, arguments);
7}
8

​ 一次只能设置一个事件监听,后续再次设置将会覆盖前者.如果想要添加多个事件处理器,则需要额外的封装.

​ 幸运的是, 2000 年 DOM 规范增加了addEventListener方法,解决了这一问题.

​ 另外,Nodejs的EventEmitter对象在Nodejs很多方面也得到了广泛的应用.

EventEmitter 对象

NodeJs内含EventEmitter类,其他对象都可以继承它,想给EventEmitter实例添加事件处理器,只需要事件类型和事件处理器函数作为参数调用on方法即可.

emitter.on('who', (name) => {
2  // balabala
3  console.log(name)
4})
5 
6// 触发
7emitter.emit('who', 'Aaron')
8

emit意味着触发,负责调用给定事件类型的所有处理器,并且传递其参数,处理器命名惯例约定为小驼峰.

EventEmitter对象所有方法为公有方法,一般约定只能从对象内部触发事件,不推荐从对象外其他地方进行调用.

PubSub 类

我们来创建一个简单PubSub类,并且实现以下几个功能:

  • on 方法: 给实例添加一个事件处理器,并且可选择是否只执行一次
  • emit 方法: 调用给定事件类型的所有处理器,并且可以传一些参数.
  • off 方法: 移除指定的事件处理器]
class PubSub {
2  constructor() {
3    this.handlers = {};
4  }
5  // 指定事件类型,处理器,以及是否为一次性处理器
6  on(eventType, handler, option = {once: false}) {
7    const self = this;
8    const once = option.once;
9    if(!(eventType in self.handlers)) {
10      self.handlers[eventType] = [];
11    }
12    // 已存在同种类型的事件和相同的处理器则直接返回,这里忽略了是否为一次性处理器
13    if(self.handlers[eventType].some(handlerItem => handlerItem.handler === handler)) return self;
14    self.handlers[eventType].push({handler, once});
15    return self;
16  }
17  // 指定事件类型和传参
18  emit(eventType, data) {
19    const self = this;
20    if(!self.handlers[eventType]) return self;
21    // 执行指定事件类型的每一个处理器
22    self.handlers[eventType].forEach(handlerItem => {
23      const {handler, once} = handlerItem;
24      handler(data);
25      // 如果是一次性处理器,则执行后自动删除
26      once && self.off(eventType, handler)
27    })
28    return self;
29  }
30  // 删除指定的事件的处理器
31  off(eventType, handler) {
32    const self = this;
33    if(!self.handlers[eventType]) return self;
34    // 更新数据存储对象,删除指定事件类型的特定处理器
35    self.handlers[eventType] = self.handlers[eventType].filter(handlerItem => handlerItem.handler !== handler);
36    return self;
37  }
38  // 额外的重置功能
39  reset() {
40    this.handlers = {};
41  }
42}
43

首先,on方法可以为指定事件类型注册处理器,并且默认提供一个option对象,其内部的once字段为布尔值,用于指定此处理器是否为一次性处理器. 每一种事件类型对应着一个数组保存着具有是否为一次性处理器标志的处理器对象.

​ 如果是首次注册则初始化一个数组保存处理器,并且跳过重复的事件与处理器组合.emit方法则对指定事件类型的所有处理器传入data参数作为处理器的参数.如果是一次性处理器,则调用之后删除此处理器.off方法用于删除指定事件类型的处理器.

​ 我们可以发现PubSub模式具有同步性,开发者可以灵活地写出bug😂,例如这段JQuery代码:

$('input[type=submit]')
2.on('click', function(){
3  $(this).trigger('click')
4})
5

​ 在添加处理器的时候显式trigger,这种同步性代码显然会引起栈溢出.

正经人谁会写出这种代码?你会吗?

​ 我们可以带有PubSub类接口的对象称之为事件化对象, 如果我们没有主动订阅事件,PubSub将是完全隐形的.不应该为某事件添加过多的事件处理器,否则可能会因为执行处理器逻辑而引起浏览器主线程阻塞,界面无响应.

​ 为了解决这个问题,我们可以对这些无需时刻发生的事件维持一个队列,设置一些异步执行逻辑或者延时任务,也许我们首次思考这个问题会写出如下示例的代码:

const tasks = [];
2setInterval(function() {
3  let nextTask;
4  if(nextTask = tasks.shift()) {
5    nextTask();
6  }
7}, 0)
8

​ 也许我们需要一个更好的方案来解决这个问题,考虑可以使用Promise来异步处理这些处理器的回调.

​ 今天的分享就到这里,PubSub模式简化了事件的命名和分发,比较著名的轮子有EventEmitter2,也许我们可以在日常的工作中使用这种模式进行开发,从而提高我们的效率,让代码更健壮优雅.
